Project 01

E-Commerce Schema Design & Data Modelling

"How should a marketplace database be structured to support analytics at scale?"

Schema DesignForeign KeysCHECK ConstraintsCTELEFT JOIN

Six tables, each with one job. Separating customers, sellers, products, orders, order_items and returns means a price change never corrupts historical data. All 9 downstream projects run off this schema without modification — designed once, queried nine ways.

Metric 1 — Realized Net Revenue by Geography

Delivered orders only. Cancelled and returned excluded. Bar chart: comparing discrete geographies where magnitude difference matters.

-- METRIC 1: REALIZED NET REVENUE BY GEOGRAPHY
select o.country, sum(oi.quantity * oi.unit_price) as net_revenue
from orders o join order_items oi on o.order_id = oi.order_id
where o.status = 'delivered'
group by o.country order by net_revenue desc;

Metric 2 — Top 5 Merchandise Demand Drivers

Includes pending orders as pipeline revenue. Excludes cancelled and returned. Horizontal bar: long product names fit on the y-axis; ranking reads top-to-bottom.

-- METRIC 2: TOP 5 MERCHANDISE DEMAND DRIVERS
select p.product_name, sum(oi.quantity*oi.unit_price) product_revenue
from order_items oi join products p on p.product_id = oi.product_id
join orders o on o.order_id = oi.order_id
where o.status <> 'cancelled' and o.status <> 'returned'
group by p.product_name order by product_revenue desc limit 5;

Metric 3 — Fulfillment Funnel Distribution

86.67% delivered, 6.67% cancelled, 6.67% returned. Doughnut: proportion of a whole split into status parts.

-- METRIC 3: FULFILLMENT FUNNEL DISTRIBUTION
select o.status, count(o.order_id) orders_count,
  round(100*count(o.order_id)/(select count(order_id) from orders)::decimal,2) percentage_of_total
from orders o group by o.status order by percentage_of_total desc;

Metric 4 — Pipeline GMV Share by Category

Electronics dominates at AED 145,451. LEFT JOIN ensures zero-revenue categories surface. Doughnut: GMV is a share-of-whole story.

-- METRIC 4: PIPELINE GMV SHARE BY CATEGORY
with x as (
  select oi.product_id, sum(oi.quantity*oi.unit_price) a
  from order_items oi join orders o on o.order_id = oi.order_id
  where o.status <> 'cancelled' and o.status <> 'returned'
  group by oi.product_id
)
select p.category_l1, coalesce(sum(x.a),0) revenue_by_category
from products p left join x on p.product_id = x.product_id
group by p.category_l1 order by revenue_by_category desc;

Project 02

Sales Performance Dashboard

"How is revenue trending across markets and categories month by month?"

date_truncGROUP BYCTEsAOVLEFT JOIN

The Monday morning questions. How did we do last month, which categories are growing, and are customers spending more or less per order. November's White Friday spike at AED 63,267 dominates the trend — the rest of the year averages around AED 9,000/month.

Metric 1 — Monthly Revenue Trend

Clear White Friday spike in November (AED 63,267 vs ~AED 9K monthly average). Line chart: time series where direction of change is the story.

-- METRIC 1: MONTHLY REVENUE TREND
select date_trunc('month', o.order_date) "month", sum(oi.quantity*oi.unit_price) revenue_by_month
from orders o join order_items oi on o.order_id = oi.order_id
where o.status='delivered'
group by date_trunc('month', o.order_date) order by "month" asc;

Metric 2 — Category Performance Over Time

Electronics dominates throughout. Fashion and Home Appliances show small but consistent presence. LEFT JOIN intentional — surfaces zero-revenue months rather than omitting them. Multi-line chart: three categories tracked on the same time axis.

-- METRIC 2: CATEGORY PERFORMANCE OVER TIME
-- left join intentional: surfaces categories with no sales in a given month
with x as (
  select oi.product_id, date_trunc('month', o.order_date) as sales_month, sum(oi.quantity*oi.unit_price) as revenue
  from orders o join order_items oi on o.order_id = oi.order_id
  where o.status = 'delivered' group by oi.product_id, date_trunc('month', o.order_date)
)
select p.category_l1, x.sales_month as "month", coalesce(sum(x.revenue), 0) as categoryrevenue_by_month
from products p left join x on p.product_id = x.product_id
group by p.category_l1, x.sales_month order by p.category_l1, "month" asc;

Metric 3 — AOV by Delivery Type

Same-day customers spend most (AED 3,359) vs standard (AED 2,435). count(distinct order_id) used — not row count — to avoid inflating AOV on multi-item orders. Bar chart: three delivery types on one value.

-- METRIC 3: AOV BY DELIVERY TYPE
with x as (select order_id, delivery_type from orders where status = 'delivered')
select x.delivery_type,
  round((sum(oi.quantity*oi.unit_price)/count(distinct x.order_id)),2) as aov
from order_items oi join x on x.order_id = oi.order_id group by x.delivery_type;

Project 03

Advanced Revenue Analytics with Window Functions

"What does cumulative GMV and month-over-month growth look like over time?"

sum() overlag()dense_rank()Window Functions

Window functions compute across rows without collapsing them — unlike GROUP BY. Running total reaches AED 157,472 by December. November's 779% MoM spike followed by -93.5% in December is the demand pull-forward signal explored further in Project 8.

Metric 1 — Running Total of Revenue

Cumulative GMV from Jan–Dec 2024. The steep climb in November reflects White Friday. Line chart: cumulative totals must always be a rising line — bars would imply each month is independent.

-- METRIC 1: RUNNING TOTAL OF REVENUE
with x as (
  select date_trunc('month', o.order_date) "month", sum(oi.quantity*oi.unit_price) revenue_by_month
  from orders o join order_items oi on o.order_id = oi.order_id
  where o.status = 'delivered' group by date_trunc('month', o.order_date)
)
select "month", sum(revenue_by_month) over (order by "month" asc) running_total_revenue
from x order by "month" asc;

Metric 2 — Month-over-Month Revenue Growth %

November: +779.32%. December: -93.52%. lag() pulls previous month's value into current row for comparison. Jan shows NULL — correct, no prior month. Bar chart: negative values drop below zero, instantly visible.

-- METRIC 2: MONTH-OVER-MONTH REVENUE GROWTH %
with x as (...)
select "month",
  round(100.0*(revenue_by_month-lag(revenue_by_month,1) over (order by "month"))
    /lag(revenue_by_month,1) over (order by "month"),2) mom_growth_percentage
from x order by "month" asc;

Metric 3 — Customer Rank by Revenue

Ahmed Al Mansoori leads at AED 37,689. dense_rank() used — ties get same rank, next rank doesn't skip. Table: names + rank + revenue in one scannable view.

-- METRIC 3: CUSTOMER RANK BY REVENUE
with x as (...), cte as (...)
select full_name, dense_rank() over (order by revenue desc) customer_rank_by_revenue, revenue
from cte order by customer_rank_by_revenue;

Project 04

Seller & Fulfillment Intelligence

"Which sellers are driving GMV, and does FBN actually outperform seller-fulfilled?"

partition byCOALESCEMulti-CTEReturn Rate

FBN generates AED 114,895 vs AED 42,577 for seller-fulfilled — but FBN has a 10.53% return rate driven entirely by Landmark Group (57.14% return rate). Samsung Gulf and Apple MENA are clean at 0% returns.

Metric 1 — Seller Revenue and Return Rate

Landmark Group is the outlier — AED 2,241 revenue with a 57.14% return rate. Two CTEs: one for delivered revenue, one for returns. LEFT JOIN ensures zero-return sellers appear with 0%, not disappear. Table: multi-column scorecard.

-- METRIC 1: SELLER REVENUE AND RETURN RATE
with x as ( -- delivered revenue per seller )
, y as ( -- return count per seller )
select x.seller_name, x.fulfillment_type, x.total_revenue,
  coalesce(y.ct,0) return_count,
  round(100.0*coalesce(y.ct,0)/(x.cnt+coalesce(y.ct,0)),2) return_pctg
from x left join y on x.seller_id = y.seller_id;

Metric 2 — FBN vs Seller-Fulfilled

FBN: AED 114,895 / 34 orders / avg rating 4.79 / 10.53% return rate. Seller-fulfilled: AED 42,577 / 18 orders / 4.09 rating / 0% returns. max(y.ct) used to avoid GROUP BY — safe since exactly one return count exists per fulfillment type. Bar: head-to-head comparison.

-- METRIC 2: FBN VS. SELLER-FULFILLED
-- max(y.ct): one return count per fulfillment type, avoids GROUP BY bloat
with y as (...)
select s.fulfillment_type, sum(...) total_revenue,
  round(avg(s.rating),2) avg_rating,
  round(100.0*coalesce(max(y.ct),0)/(...),2) return_pctg
from sellers s ... left join y on s.fulfillment_type = y.fulfillment_type
where o.status = 'delivered' group by s.fulfillment_type;

Metric 3 — Top Seller per Category

Samsung Gulf leads Electronics (AED 59,786), Noon Fashion leads Fashion (AED 4,389), Cairo Electronics leads Home Appliances (AED 5,391). dense_rank() partitioned by category resets ranking per category independently. Bar: one winner per category.

-- METRIC 3: TOP SELLER PER CATEGORY
with x as (
  select s.seller_name, p.category_l1, sum(...) revenue,
    dense_rank() over (partition by p.category_l1 order by sum(...) desc) as rank
  from order_items oi ... where o.status = 'delivered'
  group by p.category_l1, s.seller_id, s.seller_name
)
select seller_id, seller_name, category_l1, revenue
from x where rank = 1 order by category_l1;

Project 05

Customer Cohort & Retention Analytics

"Are customers coming back, and how long does it take them to return?"

Cohort Analysislead()Date Arithmetic

Every customer has purchased 4–7 times — unusually strong retention for a sample dataset. Average gap between purchases is 66.98 days, meaning customers who haven't ordered in ~134 days are at churn risk. All White Friday buyers (November) were returning customers — zero new customer acquisition during the campaign.

Metric 1 — User Retention Cohort Matrix

month_gap=0 shows acquisition size. month_gap>0 shows returning customers from that cohort. The negative month_gap (-1) in the March cohort is a data anomaly worth noting — an order placed before the recorded first_delivered_order_month, likely from the orders table including non-delivered orders in the CTE. Table: two-dimensional matrix.

-- METRIC 1: USER RETENTION COHORT MATRIX
-- month_gap=0: cohort size at acquisition | month_gap>0: returning customers
with fpd as (...), cte as (...)
select first_delivered_order_month,
  (((extract(year from order_month)-extract(year from first_delivered_order_month))*12
  +extract(month from order_month)-extract(month from first_delivered_order_month))) month_gap,
  count(distinct customer_id) cohort_size
from cte group by first_delivered_order_month, month_gap;

Metric 2 — Repeat Purchase Frequency

All customers purchased 4–7 times — no one-time buyers in this dataset. Real-world distribution would show a steep drop after 1. Bar chart: discrete frequency counts as a histogram pattern.

-- METRIC 2: REPEAT PURCHASE FREQUENCY
with x as (
  select customer_id, count(distinct order_id) purchase_frequency
  from orders where status = 'delivered' group by customer_id
)
select purchase_frequency, count(customer_id) num_customers,
  round(100.0*count(customer_id)/(select count(customer_id) from x),2) percentage_customers
from x group by purchase_frequency;

Metric 3 — Avg Days Between Purchases

66.98 days average repurchase cycle. Customers inactive for 134+ days (2×) should trigger re-engagement campaigns. lead() used — pulls next order_date into current row for each customer. KPI table: single number with business interpretation.

-- METRIC 3: AVG DAYS BETWEEN PURCHASES
with x as (
  select customer_id, order_id,
    (lead(order_date,1) over (partition by customer_id order by order_date)-order_date) days_bw_orders
  from orders where status = 'delivered'
)
select round(avg(days_bw_orders),2) avg_days_bw_orders from x;

Project 06

Returns & Refund Analysis

"Which products and categories have unacceptable return rates?"

CASE WHEN FlagRIGHT JOINFinancial Impactdense_rank()

The Adidas Hoodie has a 57.14% return rate — the only product with returns in this dataset. Fashion as a category is flagged High Risk at 33.33%. All returns are Landmark Group's products, which explains their 57.14% return rate in Project 4.

Metric 1 — Return Rate by Product

Adidas Hoodie: 57.14% return rate. All other products: 0%. RIGHT JOIN ensures all delivered products appear. ::decimal cast forces decimal division — integer division would truncate. Horizontal bar: product names on y-axis, outlier immediately visible.

-- METRIC 1: RETURN RATE BY PRODUCT
with rt_prod as (...), dlv_prod as (...)
select d.product_id, d.product_name,
  round(100*(coalesce(r.ct,0)/(d.ct+coalesce(r.ct,0))),2) rt_to_dlv_pctg
from rt_prod r right join dlv_prod d on r.product_id = d.product_id;

Metric 2 — Category Return Rate with Risk Flag

Fashion: 33.33% — High Risk. Electronics and Home Appliances: 0% — Normal. CASE WHEN flags above 10% threshold. The 10% is a business decision, not a technical one. Bar chart: category comparison with risk context.

-- METRIC 2: CATEGORY RETURN RATE + RISK FLAG
select dlv.category_l1,
  coalesce(round(100.0*coalesce(rt.ct,0)/(coalesce(rt.ct,0)+dlv.ct),2),0) rt_pctg,
  case when ... > 10 then 'High Risk' else 'Normal' end risk_status
from dlv_prod dlv left join rt_prod rt on dlv.category_l1 = rt.category_l1;

Metric 3 — Return Reason + Financial Impact

Adidas Hoodie: AED 1,245 lost to wrong size. Nike Shoes: AED 798 lost to changed mind. dense_rank() within each product finds top reason. Financial impact translates % into AED — the number operations teams act on. Table: product + reason + AED lost.

-- METRIC 3: RETURN REASONS + FINANCIAL IMPACT
with rt_reason as (
  select p.product_name, r.reason, sum(oi.unit_price*oi.quantity) revenue_lost,
    dense_rank() over (partition by p.product_name order by sum(oi.unit_price*oi.quantity) desc) reason_rank
  from products p join returns r ... join order_items oi ...
  group by p.product_name, r.reason
)
select product_name, reason, revenue_lost from rt_reason where reason_rank = 1;

Metric 4 — Platform Return Reason Distribution

Wrong size: 40%, Defective product: 40%, Changed mind: 20%. Two distinct problems — sizing (content/UX fix) and defects (supplier quality fix). Doughnut: share-of-whole reason breakdown.

-- METRIC 4: RETURN REASON DISTRIBUTION
select reason, count(reason) occurences,
  round(100.0*count(reason)/(select count(distinct return_id) from returns),2) pctg
from returns group by reason;

Project 07

Delivery & Operations Performance

"Which delivery types drive the most value, and where are orders lost to cancellations?"

Conditional AggregationMulti-Dim GROUP BYMoM by Segment

Same-day accounts for 45% of orders and AED 84,803 in revenue — the dominant delivery type. UAE same-day customers have the highest AOV at AED 5,015. Same-day also has the highest cancellation rate at 14.81% — customers who choose same-day are more likely to cancel.

Metric 1 — Order Volume & Revenue by Delivery Type

Same-day: 27 orders, AED 84,803. Next-day: 19 orders, AED 56,776. Standard: 14 orders, AED 28,529. Bar chart: three types compared on revenue magnitude.

-- METRIC 1: VOLUME & REVENUE BY DELIVERY TYPE
select o.delivery_type,
  count(distinct o.order_id) order_volume,
  round(100.0*count(distinct o.order_id)/(select count(distinct order_id) from orders),2) pctg_of_total_orders,
  sum(oi.quantity*oi.unit_price) revenue
from orders o join order_items oi on o.order_id = oi.order_id
group by o.delivery_type order by revenue desc;

Metric 2 — Cancellation Rate by Delivery Type

Same-day: 14.81% cancellation. Next-day and standard: 0%. sum(case when status='cancelled' then 1 else 0 end) — the standard conditional counting pattern. Bar chart: small differences between types visible as bar heights.

-- METRIC 2: CANCELLATION RATE BY DELIVERY TYPE
select delivery_type,
  round(100.0*sum(case when status='cancelled' then 1 else 0 end)/count(distinct order_id),2) cancellation_rate
from orders group by delivery_type;

Metric 3 — AOV by Delivery Type and Country

UAE same-day: AED 5,015 (highest). Egypt standard: AED 548 (lowest). CTE computes order-level totals first — averaging directly from order_items gives wrong results for multi-item orders. Table: 9-cell matrix (3 types × 3 countries).

-- METRIC 3: AOV BY DELIVERY TYPE AND COUNTRY
with cte as (
  select o.order_id, o.delivery_type, o.country, round(sum(oi.quantity*oi.unit_price),2) order_amount
  from order_items oi join orders o on o.order_id = oi.order_id
  group by o.order_id, o.delivery_type, o.country
)
select delivery_type, country, round(avg(order_amount),2) avg_order_value
from cte group by country, delivery_type order by country, delivery_type;

Metric 4 — MoM Order Volume Trend by Delivery Type

All three delivery types spike in November then drop sharply in December — the White Friday pull-forward signal confirmed across all delivery modes. lag() partitioned by delivery_type resets the sequence per type independently. Line breaks are intentional: same_day had no orders in July and standard had no orders in August and November — the SQL query returns no row for those months, so no MoM growth can be calculated. Filling those gaps with 0% would imply orders were flat, which is factually incorrect. A break is the honest representation of missing data. Multi-line: three trend lines on the same time axis.

-- METRIC 4: MoM ORDER VOLUME TREND BY DELIVERY TYPE
with counts as (
  select delivery_type, date_trunc('month', order_date) order_month,
    count(distinct order_id) num_orders,
    lag(count(distinct order_id),1) over (partition by delivery_type order by date_trunc('month',order_date)) lags
  from orders group by delivery_type, date_trunc('month', order_date)
)
select delivery_type, order_month, round(100.0*(num_orders-lags)/lags,2) MoM_growth
from counts order by delivery_type, order_month;

Project 08

Marketing & Campaign Effectiveness

"Did White Friday drive incremental revenue or just pull demand forward from December?"

Period LabellingCampaign UpliftPull-Forward SignalTwo-Level CTE

The verdict: strong pull-forward signal. Pre-campaign average was ~AED 9K/month. November hit AED 63,267. December dropped to AED 4,097 — below even October's AED 7,195. December fell below baseline, which means White Friday pulled forward demand that wasn't replaced.

Metric 1 — Pre / During / Post Campaign Revenue

Pre-campaign AED 90,108 (31 orders over 10 months = ~AED 9K/month avg). Campaign: AED 63,267 in one month. Post: AED 4,097. The ORDER BY uses a nested CASE to force chronological display. Bar chart: three periods, one metric — the contrast is the story.

-- METRIC 1: PRE / DURING / POST CAMPAIGN REVENUE
select
  case when o.order_date < '2024-11-01' then 'pre_campaign'
       when o.order_date > '2024-11-30' then 'post_campaign'
       else 'campaign' end sales_period,
  sum(oi.quantity*oi.unit_price) revenue, count(distinct o.order_id) order_volume
from orders o join order_items oi on o.order_id = oi.order_id
where o.status = 'delivered' group by sales_period;

Metric 2 — Campaign Uplift by Category

Electronics: +579% uplift. Home Appliances: +275%. Fashion: +218%. All categories benefited — Electronics most. Baseline = avg monthly revenue excluding November. Bar chart: positive uplift bars per category.

-- METRIC 2: CAMPAIGN UPLIFT BY CATEGORY
with non_campaign_average as (...),
     campaign_data as (...)
select cd.category_l1,
  round(100.0*(cd.revenue-nca.avg_revenue)/nca.avg_revenue,2) uplift_pctg
from non_campaign_average nca
join campaign_data cd on nca.category_l1 = cd.category_l1;

Metric 3 — New vs Returning During Campaign

100% of November buyers were returning customers — zero new customer acquisition during White Friday. This means the campaign spent budget rewarding existing customers, not acquiring new ones. Doughnut: new vs repeat as share of campaign buyers.

-- METRIC 3: NEW VS RETURNING DURING CAMPAIGN
with pre_camp_cust as (
  select distinct customer_id from orders where order_date < '2024-11-01'
), tagged_cust as (
  select distinct o.customer_id,
    case when pcc.customer_id is null then 'new' else 'repeat' end customer_type
  from orders o left join pre_camp_cust pcc on o.customer_id = pcc.customer_id
  where o.order_date between '2024-11-01' and '2024-11-30'
)
select customer_type, count(customer_id) num, ... from tagged_cust group by customer_type;

Metric 4 — Demand Pull-Forward Signal

Oct: AED 7,195. Nov: AED 63,267 (+779%). Dec: AED 4,097 (-93.5%). December fell below October baseline — confirmed pull-forward. September included in CTE to give October a lag value, then filtered in outer query after window functions run. Dual line: revenue and MoM growth on same axis.

-- METRIC 4: PULL-FORWARD SIGNAL (two-level CTE pattern)
-- Sep in CTE feeds Oct's lag; filtered in outer query after window runs
with cte as (...), with_lag as (
  select order_month, revenue_by_month,
    round((100.0*(revenue_by_month-lag(revenue_by_month,1) over (order by order_month))
      /lag(revenue_by_month,1) over (order by order_month)),2) revenue_growth,
    ...volume_growth
  from cte
)
select * from with_lag where order_month <> '2024-09-01';

Executive GMV Dashboard

"What is the full health of this business in a single query set?"

Total GMV: AED 170,108. Delivered revenue: AED 157,472 (92.6% realisation rate). 3 High-value customers generate AED 88,878 — more than the other 7 combined. UAE × Electronics dominates the country-category matrix at AED 113,371.

Metric 1 — Business Health Summary

Total GMV AED 170,108 · Delivered AED 157,472 · Cancellation 6.67% · Return rate 6.67% · 10 customers · 7 sellers. count(distinct case when status='cancelled' then order_id end) pattern — NULL ignored by COUNT, no subquery needed. Table: single-row KPI summary.

-- METRIC 1: BUSINESS HEALTH SUMMARY
select
  sum(oi.quantity*oi.unit_price) total_gmv,
  sum(case when o.status='delivered' then oi.quantity*oi.unit_price else 0 end) delivered_revenue,
  round(100.0*count(distinct case when o.status='cancelled' then o.order_id end)/count(distinct o.order_id),2) cancellation_rate,
  round(100.0*count(distinct case when o.status='returned' then o.order_id end)/count(distinct o.order_id),2) return_rate,
  count(distinct o.customer_id) active_customers, count(distinct p.seller_id) active_sellers
from orders o join order_items oi ... join products p ...;

Metric 2 — Rolling 3-Month Revenue Average

The rolling average peaks at AED 26,386 in November vs actual AED 63,267 — showing the spike is a one-time event. rows between 2 preceding and current row defines a 3-month trailing window. Dual line: actual vs smoothed trend on same axis.

-- METRIC 2: ROLLING 3-MONTH REVENUE AVERAGE
select order_month, revenue,
  round(avg(revenue) over (order by order_month rows between 2 preceding and current row),2) three_mth_rolling_avg
from monthly_revenue;

Metric 3 — Country × Category GMV Matrix

UAE × Electronics: AED 113,371 (dominant). KSA × Home Appliances and UAE × Home Appliances both zero — no home appliance sellers in those markets. CROSS JOIN generates all 15 combinations, LEFT JOIN fills with 0 where no sales exist. Table: 15-cell matrix.

-- METRIC 3: COUNTRY x CATEGORY GMV MATRIX
-- cross join ensures every combination appears even with zero GMV
with countries as (select distinct country from orders),
  categories as (select distinct category_l1 from products),
  matrix as (select co.country, ca.category_l1 from countries co cross join categories ca),
  answer as (...)
select m.country, m.category_l1, coalesce(a.gmv,0.00) gmv
from matrix m left join answer a on m.country=a.country and m.category_l1=a.category_l1;

Metric 4 — Full Seller Scorecard

Seller 7 (Landmark Group, FBN) ranks 4th in FBN with a 57.14% return rate — a quality flag despite being FBN. dense_rank() partitioned by fulfillment_type ranks within each group independently. Table: 7-column scorecard per seller.

-- METRIC 4: FULL SELLER SCORECARD
select s.seller_id, s.fulfillment_type,
  sum(case when o.status='delivered' then oi.quantity*oi.unit_price else 0 end) total_revenue,
  round(100.0*count(distinct case when o.status='returned' then o.order_id else null end)/count(distinct o.order_id),2) return_rate,
  round(s.rating,2) average_rating, count(distinct o.order_id) order_count,
  dense_rank() over (partition by fulfillment_type order by sum(...) desc) rank_by_fulfillment_type
from sellers s ... group by s.seller_id, s.fulfillment_type, s.rating;

Metric 5 — Customer LTV Segments

High (≥AED 20K): 3 customers, AED 88,878. Mid: 4 customers, AED 53,074. Low (<AED 10K): 3 customers, AED 15,520. The 3 High-value customers generate more revenue than the other 7 combined. Doughnut: revenue share by LTV segment.

-- METRIC 5: CUSTOMER LTV SEGMENTS
with cte as (
  select o.customer_id,
    sum(case when o.status='delivered' then oi.quantity*oi.unit_price else 0 end) revenue,
    case when sum(...) < 10000 then 'Low'
         when sum(...) >= 20000 then 'High' else 'Mid' end segment
  from orders o join order_items oi ... group by o.customer_id
)
select segment, sum(revenue), count(customer_id), round(avg(revenue),2)
from cte group by segment;
